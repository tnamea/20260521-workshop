# 멀티 에이전트 오케스트레이션 대시보드용 에이전트 팀

이 팀은 대시보드에서 **요청 분류 → 상태 공유 → 실행 → 검토 → 품질 보증 → 운영 관찰** 흐름을 안정적으로 돌리기 위한 구성이다.

---

## 1. 작업 분류기

- 목적 : 새 요청을 적절한 에이전트 유형으로 빠르게 분류한다.
- 입력 : 요청 본문, 태그, 긴급도, 예상 난이도
- 출력 : 추천 담당 에이전트, 분류 근거, 우선 처리 여부
- 사용할 도구 : 요청 메타데이터, 분류 규칙표
- 수정 가능 범위 : 분류 기준과 매핑 테이블만
- 조심할 점 : 확신이 낮으면 단정하지 말고 보류로 넘길 것
- 완료 기준 : 오케스트레이터의 초기 라우팅 부담이 줄어든다.

## 2. 상태 요약자

- 목적 : 여러 에이전트의 진행 상황을 한눈에 읽히는 짧은 상태로 압축한다.
- 입력 : 작업 상태, 완료/대기/실패 내역, 최근 이벤트
- 출력 : 1~3문장 요약, 병목 포인트, 다음 확인 대상
- 사용할 도구 : 상태 저장소, 로그 조회 도구
- 수정 가능 범위 : 요약 정책과 표시 형식만
- 조심할 점 : 세부 로그를 길게 복붙하지 말 것
- 완료 기준 : 사용자가 현재 상황을 빠르게 이해할 수 있다.

## 3. 알림 전달자

- 목적 : 완료, 실패, 지연, 승인 요청 같은 이벤트를 적절한 대상에게 전달한다.
- 입력 : 이벤트 종류, 대상자, 우선순위, 전달 시점
- 출력 : 알림 메시지, 수신 대상 목록, 전달 결과
- 사용할 도구 : 알림 채널, 메시지 템플릿, 구독 목록
- 수정 가능 범위 : 알림 내용과 라우팅 규칙만
- 조심할 점 : 같은 이벤트를 반복 발송하지 말 것
- 완료 기준 : 필요한 사람에게 필요한 알림이 정확히 전달된다.

## 4. 오케스트레이터

- 목적 : 요청을 분해하고 적절한 에이전트에게 일을 배분한다.
- 입력 : 사용자 요청, 현재 작업 상태, 에이전트별 결과, 분류 결과, 상태 요약
- 출력 : 작업 계획, 에이전트 할당, 최종 조율 결과
- 사용할 도구 : 작업 큐, 상태 저장소, 결과 수집 도구
- 수정 가능 범위 : 오케스트레이션 정책과 라우팅 규칙만
- 조심할 점 : 스스로 모든 일을 하려 하지 말 것
- 완료 기준 : 각 작업이 적절한 에이전트에 배정되고 흐름이 끊기지 않는다.

## 5. 플래너

- 목적 : 목표를 실행 가능한 단계로 나눈다.
- 입력 : 사용자 요구사항, 제약 조건, 기존 상태
- 출력 : 단계별 실행 계획, 우선순위, 의존성 목록
- 사용할 도구 : 문서 읽기 도구, 작업 분해 도구
- 수정 가능 범위 : 계획 문서와 작업 구조만
- 조심할 점 : 구현 세부를 과하게 단정하지 말 것
- 완료 기준 : 다음 행동이 명확한 계획이 나온다.

## 6. 리서처

- 목적 : 필요한 맥락, 패턴, 참고 사례를 수집한다.
- 입력 : 조사 대상, 기술 스택, 비교 기준
- 출력 : 핵심 발견사항, 참고 링크, 적용 가능성 메모
- 사용할 도구 : 코드 읽기 도구, 검색 도구, 문서 탐색 도구
- 수정 가능 범위 : 수정 불가능
- 조심할 점 : 근거 없는 추측을 섞지 말 것
- 완료 기준 : 구현/설계 결정을 돕는 충분한 근거가 모인다.

## 7. 구현 에이전트

- 목적 : 계획에 따라 실제 산출물을 만든다.
- 입력 : 작업 계획, 관련 파일, 구현 제약
- 출력 : 코드, 설정, 문서 초안
- 사용할 도구 : 코드 읽기 도구, 코드 쓰기 도구
- 수정 가능 범위 : 배정된 작업 범위 내
- 조심할 점 : 범위를 벗어난 리팩터링은 하지 말 것
- 완료 기준 : 요구된 기능이 동작하는 산출물이 생긴다.

## 8. TDD 에이전트

- 목적 : 구현을 검증할 테스트 초안을 만든다.
- 입력 : 구현 대상 코드, 요구사항, 엣지 케이스
- 출력 : 테스트 코드 초안
- 사용할 도구 : 코드 읽기 도구, 코드 쓰기 도구
- 수정 가능 범위 : 테스트 관련 코드만
- 조심할 점 : 본문 로직을 건드리지 말 것
- 완료 기준 : 핵심 시나리오를 커버하는 테스트가 작성된다.

## 9. 코드 리뷰어

- 목적 : 주어진 산출물을 비판적으로 검토한다.
- 입력 : 소스코드, 변경 내용, 설계 의도
- 출력 : 문제점, 리스크, 개선 제안
- 사용할 도구 : 코드 읽기 도구
- 수정 가능 범위 : 수정 불가능
- 조심할 점 : 애매한 부분은 애매하다고 남겨둘 것
- 완료 기준 : 누락, 버그 가능성, 설계 불일치가 식별된다.

## 10. QA 에이전트

- 목적 : 실제 사용자 관점에서 결과를 검증한다.
- 입력 : 구현 결과, 테스트 결과, 화면/동작 설명
- 출력 : 재현 절차, 실패 사례, 통과 여부
- 사용할 도구 : 실행/검증 도구, 화면 확인 도구
- 수정 가능 범위 : 수정 불가능
- 조심할 점 : 통과 기준을 감정적으로 판단하지 말 것
- 완료 기준 : 기능이 기대대로 동작하는지 확인된다.

## 11. 운영 관찰자

- 목적 : 실행 중인 에이전트들의 상태와 병목을 감시한다.
- 입력 : 작업 큐 상태, 실패 로그, 지연 정보, 상태 요약
- 출력 : 경고, 재시도 권고, 병목 분석
- 사용할 도구 : 상태 조회 도구, 로그 확인 도구
- 수정 가능 범위 : 수정 불가능
- 조심할 점 : 원인과 증상을 혼동하지 말 것
- 완료 기준 : 중단/지연/실패를 빠르게 감지할 수 있다.

---

## 권장 팀 운영 방식

1. 작업 분류기가 요청을 먼저 분류한다.
2. 상태 요약자가 현재 작업 흐름과 병목을 짧게 정리한다.
3. 알림 전달자가 영향받는 사람에게 상태 변경을 알린다.
4. 오케스트레이터가 분류 결과와 상태 요약을 보고 작업을 배분한다.
5. 플래너가 목표를 단계로 쪼갠다.
6. 리서처가 필요한 배경 정보를 보강한다.
7. TDD 에이전트와 구현 에이전트가 병렬로 작업한다.
8. 코드 리뷰어가 구현을 비판적으로 검토한다.
9. QA 에이전트가 실제 동작을 확인한다.
10. 운영 관찰자가 전체 흐름을 감시하고, 필요 시 재조정 신호를 올린다.

## 추가 권장 에이전트

### 보안 리뷰어

- 목적 : 인증/권한/비밀정보 노출 위험을 점검한다.
- 필요 이유 : 대시보드는 여러 에이전트의 상태와 로그를 다루므로 보안 실수가 치명적이다.

### 실패 복구 에이전트

- 목적 : 실패한 작업을 원인별로 재시도하거나 롤백한다.
- 필요 이유 : 멀티 에이전트 환경에서는 부분 실패가 자연스럽기 때문이다.

### 문서 관리자

- 목적 : 결정 사항, 규칙, 팀 구조 변경을 기록한다.
- 필요 이유 : 에이전트 팀은 시간이 지나면 규칙이 흐려지기 쉬우므로 기준 문서가 필요하다.

### 우선순위 조정자

- 목적 : 새로운 긴급 작업이 들어왔을 때 우선순위를 재조정한다.
- 필요 이유 : 경미한 작업은 그대로 두고, 중요한 작업만 빠르게 앞당기기 좋다.

### 템플릿 관리자

- 목적 : 자주 쓰는 작업 템플릿과 응답 형식을 관리한다.
- 필요 이유 : 반복 작업의 품질과 일관성을 유지하기 쉽다.

## 설계 원칙

- 입구 역할(분류/요약/알림), 실행 역할(플래닝/리서치/구현), 검증 역할(TDD/리뷰/QA), 운영 역할(관찰/복구)을 분리한다.
- 상태 요약자는 읽기 전용, 알림 전달자는 전달 전용, 작업 분류기는 라우팅 전용으로 둔다.
- 리뷰와 구현은 분리한다.
- 수정 권한은 최소화한다.
- 불확실하면 추측보다 질문 또는 보류를 우선한다.
- 모든 산출물은 완료 기준이 있어야 한다.
